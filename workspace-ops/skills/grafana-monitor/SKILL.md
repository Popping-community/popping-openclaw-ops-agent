# Grafana Monitor Skill

PoppingOps 서버 상태 보고 스킬. 기본은 Railway 로컬의 health-check snapshot을 사용하고, 사용자가 "실시간"을 명시하면 EC2에서 현재 gauge 메트릭을 SSH로 다시 수집한다.

## Modes

### Snapshot mode

Trigger:
- "서버 상태 확인해줘", "메트릭 확인해줘", "서버 상태", "대시보드"

Use the latest Railway-local snapshot generated by `/scripts/health-check.sh`, and calculate snapshot freshness before reporting:

```bash
SNAPSHOT_FILE="/tmp/health-check-alerts/status-current.env"
print_snapshot_with_freshness() {
  if [ ! -f "$SNAPSHOT_FILE" ]; then
    echo "snapshot_unavailable=true"
    return
  fi

  cat "$SNAPSHOT_FILE"
  collected_at_epoch=$(grep '^collected_at_epoch=' "$SNAPSHOT_FILE" | cut -d= -f2 | tr -d '"')
  now_epoch=$(date +%s)
  if [ -n "$collected_at_epoch" ]; then
    snapshot_age_min=$(( (now_epoch - collected_at_epoch) / 60 ))
    echo "snapshot_age_min=\"$snapshot_age_min\""
    if [ "$snapshot_age_min" -ge 90 ]; then
      echo 'snapshot_freshness_status="CRITICAL"'
      echo 'snapshot_freshness_label="수집 중단 가능성"'
    elif [ "$snapshot_age_min" -ge 45 ]; then
      echo 'snapshot_freshness_status="WARN"'
      echo 'snapshot_freshness_label="오래됨"'
    else
      echo 'snapshot_freshness_status="OK"'
      echo 'snapshot_freshness_label="최신"'
    fi
  else
    echo 'snapshot_freshness_status="WARN"'
    echo 'snapshot_freshness_label="측정시각 누락"'
  fi
}
print_snapshot_with_freshness
```

Report `collected_at_kst` as the snapshot measurement time. RPS, response time, error rate, MySQL QPS, and slow query delta are recent 30-minute average/delta values from this snapshot. If `rate_status_*` is `init`, `missing`, or `reset`, report the rate metric as "계산 대기" or "수집 실패" instead of guessing.

Snapshot freshness thresholds:
- `snapshot_age_min < 45`: `OK`
- `45 <= snapshot_age_min < 90`: `WARN`, report data status as `오래됨, {snapshot_age_min}분 전 snapshot`
- `snapshot_age_min >= 90`: `CRITICAL`, report data status as `수집 중단 가능성, {snapshot_age_min}분 전 snapshot`

### Realtime mode

Trigger:
- "실시간 서버 상태 확인해줘", "지금 바로 서버 상태 확인해줘", "실시간 메트릭 확인"

Use realtime SSH collection for current gauge metrics, and still use the latest snapshot for rate metrics:

```bash
echo "=== LOCAL RATE SNAPSHOT ==="
SNAPSHOT_FILE="/tmp/health-check-alerts/status-current.env"
print_snapshot_with_freshness() {
  if [ ! -f "$SNAPSHOT_FILE" ]; then
    echo "snapshot_unavailable=true"
    return
  fi

  cat "$SNAPSHOT_FILE"
  collected_at_epoch=$(grep '^collected_at_epoch=' "$SNAPSHOT_FILE" | cut -d= -f2 | tr -d '"')
  now_epoch=$(date +%s)
  if [ -n "$collected_at_epoch" ]; then
    snapshot_age_min=$(( (now_epoch - collected_at_epoch) / 60 ))
    echo "snapshot_age_min=\"$snapshot_age_min\""
    if [ "$snapshot_age_min" -ge 90 ]; then
      echo 'snapshot_freshness_status="CRITICAL"'
      echo 'snapshot_freshness_label="수집 중단 가능성"'
    elif [ "$snapshot_age_min" -ge 45 ]; then
      echo 'snapshot_freshness_status="WARN"'
      echo 'snapshot_freshness_label="오래됨"'
    else
      echo 'snapshot_freshness_status="OK"'
      echo 'snapshot_freshness_label="최신"'
    fi
  else
    echo 'snapshot_freshness_status="WARN"'
    echo 'snapshot_freshness_label="측정시각 누락"'
  fi
}
print_snapshot_with_freshness

echo ""
echo "=== REALTIME EC2 GAUGES ==="
EC2_HOST="${EC2_HOST:-52.79.56.222}"
EC2_SSH_PORT="${EC2_SSH_PORT:-2222}"
EC2_SSH_USER="${EC2_SSH_USER:-ec2-user}"
APP_ACTUATOR_PORT="${APP_ACTUATOR_PORT:-8081}"
NODE_EXPORTER_PORT="${NODE_EXPORTER_PORT:-9100}"
MYSQL_EXPORTER_PORT="${MYSQL_EXPORTER_PORT:-9104}"
ssh -i /root/.ssh/ec2-key.pem -o StrictHostKeyChecking=no -p "$EC2_SSH_PORT" -o ConnectTimeout=5 "${EC2_SSH_USER}@${EC2_HOST}" \
  "APP_ACTUATOR_PORT=${APP_ACTUATOR_PORT} NODE_EXPORTER_PORT=${NODE_EXPORTER_PORT} MYSQL_EXPORTER_PORT=${MYSQL_EXPORTER_PORT} bash -s" <<'REMOTE_SCRIPT'
echo "realtime_collected_at_utc=$(date -u "+%Y-%m-%dT%H:%M:%S+00:00")"
echo "realtime_collected_at_kst=$(date -u -d "+9 hours" "+%Y-%m-%d %H:%M KST")"

echo "=== ACTUATOR HEALTH ==="
curl -s "http://localhost:${APP_ACTUATOR_PORT}/actuator/health"

echo "=== SPRING BOOT GAUGES ==="
curl -s "http://localhost:${APP_ACTUATOR_PORT}/actuator/prometheus" | grep -E "^(tomcat_threads|hikaricp_connections_(active|max)|jvm_memory_(used|max)_bytes.*heap|jvm_gc_pause_seconds_sum)" | grep -v "#"

echo "=== NODE GAUGES ==="
curl -s "http://localhost:${NODE_EXPORTER_PORT}/metrics" | grep -E "^(node_memory_(MemAvailable|MemTotal|SwapFree|SwapTotal)_bytes |node_load(1|5|15) |node_filesystem_(avail|size)_bytes.*fstype=\"xfs\"|node_time_seconds |node_boot_time_seconds )" | grep -v "#" | grep -v tmpfs

echo "=== MYSQL GAUGES ==="
curl -s "http://localhost:${MYSQL_EXPORTER_PORT}/metrics" | grep -E "^mysql_global_(status_threads_connected|variables_max_connections|status_queries|status_slow_queries|status_table_locks_waited) " | grep -v "#"
REMOTE_SCRIPT
```

Report both timestamps:
- `현재값 측정시각`: `realtime_collected_at_kst`
- `트래픽 지표 기준`: snapshot `collected_at_kst`, recent `snapshot_interval_min` minutes
- `트래픽 데이터 상태`: snapshot freshness status and age

## Reporting Rules

- Always include the measurement time section first.
- Snapshot reports must say: `데이터: health-check 스냅샷`, `측정시각: {collected_at_kst}`, `데이터 상태: {snapshot_freshness_label}, {snapshot_age_min}분 전 snapshot`, `수집주기: {snapshot_interval_min}분`.
- Realtime reports must say: `현재값 측정시각: {realtime_collected_at_kst}` and `트래픽 지표 기준: {collected_at_kst} 스냅샷, 최근 {snapshot_interval_min}분 평균, 데이터 상태 {snapshot_freshness_label}`.
- If `snapshot_freshness_status` is `WARN`, raise the overall severity to at least `WARN`.
- If `snapshot_freshness_status` is `CRITICAL`, raise the overall severity to `CRITICAL` and recommend checking whether `health-check.sh` is still running.
- If the snapshot is missing, stale, or has `rate_status_* != ok`, clearly report the affected metric as unavailable.
- Counter totals are cumulative. Rate metrics are interval values from snapshot delta calculation.
- Recommended actions must be runbook-first. Use `/root/.openclaw/docs/runbook.md` when WARN/CRITICAL status appears or the user asks what to do next.
- Runbook-based recommendations are guidance, not automatic actions. Do not execute restart, config changes, deploy, or write operations unless the user explicitly approves.
- If no runbook section matches, clearly separate fallback advice under `추론 기반 권장 확인`.

## Severity Thresholds

| Metric | WARN | CRITICAL |
|--------|------|----------|
| CPU Load (1 vCPU) | Load1 > 1.0 | Load1 > 2.0 |
| Memory | > 80% | > 95% |
| Swap | > 50% | > 80% |
| JVM Heap | > 80% | > 95% |
| DB Connection (HikariCP) | > 70% | > 90% |
| Tomcat Thread | > 70% | > 90% |
| Disk Usage | > 80% | > 90% |
| Avg Response | > 1s | > 3s |
| Error Rate | > 1% | > 5% |

Overall severity = highest individual severity.

## Runbook Mapping

| Condition | Runbook section |
|-----------|-----------------|
| Memory WARN | `메모리 WARN 대응` |
| Memory CRITICAL | `메모리 CRITICAL 대응` |
| SSH failure | `SSH 실패 대응` |
| Spring Boot Health DOWN | `Spring Boot Health DOWN 대응` |
| Webhook failure | `Webhook 알림 실패 대응` |
| Snapshot stale or missing | `Snapshot Stale 대응` |
| Avg Response or Error Rate WARN/CRITICAL | `트래픽 알림 대응` |
| Recovery check | `복구 확인` |

Fallback rules:
- If no runbook section matches the issue, say `runbook에 직접 절차 없음`.
- Then provide conservative read-only diagnostics under `추론 기반 권장 확인`.
- Allowed fallback actions: logs, metrics, status checks, timestamp checks, cross-validation.
- Do not recommend restart, delete, config changes, deploy, or write operations unless the user explicitly approves.
- If the same pattern repeats, suggest adding it to `/root/.openclaw/docs/runbook.md`.

Execution rule:
- Runbook-based recommendations are still guidance.
- If a runbook suggests restart or another state-changing operation, present it as `승인 후 검토할 조치` and wait for explicit user approval before execution.
